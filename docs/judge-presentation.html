02 · Our Thinking

We Started With One Question

What would a world-class senior collector do if they had unlimited time and perfect memory? That's what we automated.

The "Senior Analyst" Mental Model

A senior analyst reviewing an overdue invoice doesn't just look at DPD. They think: "Who is this borrower? Have they been late before? Did they respond to reminders? What's their overall exposure with us? Are there any external red flags — news, compliance issues? What did we try last time, and did it work?" Then they make a judgment call on the right next action. We built exactly that — in software.

Each layer feeds the next. Behavior signals improve delay prediction accuracy. Delay probabilities drive collection strategy urgency. Probabilities × amounts = realistic cash flow outlook. The agent synthesizes everything into a plain-English recommendation.

Three Core Beliefs That Shaped Our Architecture

🎯 Predict, Don't React

Every invoice should have a delay_probability computed before it's overdue — so we act on future risk, not past buckets. ML models trained on 1,500+ INR records power this.

🧠 Behavior Is Evidence

Payment history, on-time ratio, trend direction, PTP adherence — these behavioral signals are more predictive than invoice amount or DPD alone. We built a full Behavior Engine to extract this.

⚡ Agent, Not Dashboard

A dashboard shows data. An agent thinks and acts. Our GPT-4o agent with 9 tools decides its own investigation sequence, calls the right services in order, and writes a business-readable recommendation.

03 · What We Built

5 Intelligence Pillars — Each Feeding the Next

Not isolated modules — a connected pipeline where every layer's output becomes the next layer's input.

Pillar 01
Payment Behavior Engine

Analyzes a borrower's historical payment personality — not just credit score. Computes on-time ratio, average delay, deterioration trend, followup dependency, partial payment frequency. Classifies borrower as Chronic Delayed Payer, Reminder-Driven, Reliable Payer, etc.

behavior_type: "Chronic Delayed Payer"
on_time_ratio: 31% | avg_delay_days: 12
trend: "Worsening" | nach_recommended: true
Pillar 02
Delay Prediction Engine

XGBoost binary classifier trained on 1,500 INR invoices. Features: invoice amount, DPD, credit score, historical delay, behavior signals. Returns delay probability, risk score, risk tier, SHAP-style top drivers, model confidence, evidence score, and fallback flag.

delay_probability: 0.82 | risk_tier: "High"
confidence: 0.71 | evidence_score: 0.67
top_drivers: ["45 DPD", "Chronic payer", "₹42.5L invoice"]
Pillar 03
Collection Strategy Engine

Generates 3 candidate actions per urgency tier (aggressive / structured / relationship-based). An agentic selector scores each on recovery estimate × cost × urgency weight, picks the winner, and writes a selection rationale. Priority score formula: delay_prob × 60 + amount bonus + DPD bonus + behavior bonus.

recommended_action: "Payment Plan Negotiation"
urgency: "High" | priority_score: 87/100
3 candidates evaluated | selection_score: 0.82
Pillar 04
Cash Flow Forecasting Engine

Probability-weighted cashflow forecast (not optimistic "everyone pays" assumption). Expected collections = Σ(amount × pay_probability) per horizon. Computes amount_at_risk, shortfall_signal, and 7/30-day expected inflows. Overdue carry-forward included.

expected_7_day: ₹42.5L | expected_30_day: ₹1.8Cr
amount_at_risk: ₹1.94Cr | shortfall_signal: true
Pillar 05
Agentic AI Layer — GPT-4o with 9 Tools (ReAct Loop)

The crown of the platform. GPT-4o autonomously decides which tools to call, in what order, based on what it discovers at each step. It reasons → acts → observes → reasons again. Every tool call is visible in the UI as a step-by-step trace with inputs, outputs, and the agent's thought process.

1. analyze_payment_behavior 2. predict_invoice_delay 3. optimize_collection_strategy 4. get_borrower_risk 5. get_portfolio_summary 6. get_invoice_details 7. check_external_signals 8. get_interaction_history 9. get_borrower_enrichment
04 · Three Innovation Layers We Added

Beyond the Basic Build — This Is Where We Go Deeper

Most hackathon projects stop at "predict + show dashboard." We added three layers that make this production-realistic.

🛰️ Sentinel External Signals Engine

Monitors external risk signals per borrower — leadership changes, NCLT proceedings, GST suspension, AP contact failures, email anomalies, sector distress. Maps to the CredCheck/data mapping doc's "MCA, legal, and bureau" enrichment family.

🔴 NCLT insolvency proceedings (Economic Times)

📜 Interaction History + Feedback Loop

Every collection touchpoint is recorded — calls, emails, PTPs, broken PTPs, legal notices, field visits. The system computes per-borrower action effectiveness from past outcomes. This is the feedback loop: the system learns from its own history.

Legal Notice: 100% success rate (1/1)
Call: 33% (1/3) | Email: 0% (0/1)
→ Recommend: Legal Notice first, not email
🏛️ CredCheck Enrichment Layer

MCA compliance score, GST filing health, EPFO workforce stability, bureau credit health (CIBIL), legal case summary. Explicit data availability flags: has_bureau_data, has_gst_data, has_legal_data. Maps directly to the CredCheck data doc.

07 · Evidence Confidence Layer

We Don't Show All Predictions As Equally Certain

The data mapping document identified this as a "Must-have for maturity." We built it. The system explicitly models how much it knows and tells the user when to trust the output less.

Model Confidence Score

Returned by the ML model itself. If below 0.55 threshold, system automatically falls back to the rule engine and flags used_fallback: true in the response.

Evidence Score

Fraction of optional enrichment fields provided (behavior signals, trend data, historical context). 0/6 fields = 0%, 6/6 = 100%. Shown as a separate bar in the UI.

History Confidence Boost

Each recorded interaction adds +4% confidence (capped at +25%). A borrower with 6 interactions gets 24% boost — the system has learned from their specific patterns.

⚠️ Low Confidence Example

Scenario: New borrower, no payment history, no behavior data, no CredCheck report yet.

Model confidence: 45% | Evidence score: 0% | Missing data: behavior_type, on_time_ratio, avg_delay_historical, trend...

→ "Rule Fallback" badge shown. Confidence bar red. Missing data chips displayed. User knows to treat prediction cautiously.

✅ High Confidence Example

Scenario: TechNova — full behavior profile, bureau data, 6 interaction records, CredCheck enrichment.

Model confidence: 82% | Evidence score: 83% | History boost: +24% | No missing data

→ Green confidence bar. All data flags present. Agent recommendation is fully evidenced. Manager can act with confidence.

Key Talking Points for Judges

🎯 "Why is this not just a dashboard?"

It doesn't show data — it makes decisions. The system tells you which invoice to call first, what to say, which channel to use, and why. The AI agent reasons across 9 data sources autonomously. You can ask it anything in plain English.

🔄 "How does it learn?"

Interaction history feeds back into confidence scoring and action recommendations. After 6 failed calls for TechNova, the system stops recommending calls and recommends Field Visit — because that's the only thing that worked historically.

🛡️ "How trustworthy are the predictions?"

Every prediction comes with a confidence score, evidence score, missing-data indicators, and a fallback flag. We never show predictions as equally certain — a sparse-data borrower gets explicit warnings. This is the Confidence Layer.

📊 "Can this scale to real data?"

Yes. The backend is service-oriented — each pillar is independent. Replace mock data with DB queries. Models retrain on real INR transaction data. The Docker setup is production-ready. Real CredCheck/Digitap APIs slot in with config changes.
